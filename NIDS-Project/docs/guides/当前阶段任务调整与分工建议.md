# 当前阶段任务调整与分工建议

适用时间：2026年4月20日以后  
适用范围：当前项目收尾、中期检查展示、系统演示与核心功能补强  
不包含内容：软著申请、论文撰写

## 一、先说结论

你现在这几份“14天冲刺”文档，方向有价值，但整体上 **偏理想化、偏发散，不够贴合仓库现状**，需要收缩。

当前更合理的做法不是同时强推：

1. 从 NSL-KDD 直接切到 CICIDS2017 做主线
2. 在 14 天内补齐 CNN-LSTM、Ensemble、PCAP 真分析、前后端联调
3. 还把 TD3、INT8、DPDK、Transformer 这些高级词继续挂在界面和文档里

这样安排的问题是：任务彼此强耦合，且很多前置条件还没补齐，最后容易出现“文档写完了、代码也有文件名，但真正可验收的闭环并没有稳定跑通”。

## 二、当前仓库的真实状态

结合本地仓库和 GitHub 已跟踪内容，当前项目更接近下面这个状态：

1. 主线成果仍然是 **NSL-KDD + RandomForest + Flask + 前端展示**。
2. `CICIDS2017` 相关脚本已经新建，但本地并没有看到对应数据资产落地，且脚本路径体系和当前项目目录并不统一。
3. `PCAP` 上传接口已经有壳，但还没有真正的分析逻辑。
4. 深度学习脚本已经开始写，但依赖、数据路径、模型接入方式还没有完全统一。
5. 前端能演示，但界面文案明显超前于后端实际能力。
6. GitHub 远端的主要内容还是原有主线；你新写的几份冲刺文档目前更多是本地临时稿，不是已经沉淀好的正式协作文档。

## 三、调整原则

当前阶段建议按下面四条原则收任务：

1. **先保闭环，再做扩展。**  
   优先保证“数据预处理 -> 模型训练 -> 后端接口 -> 前端展示 -> 演示脚本”这一条线稳定。

2. **先保 NSL-KDD 主线，再开 CICIDS2017 支线。**  
   NSL-KDD 是当前已有成果的依据，CICIDS2017 可以做加分项，不要反客为主。

3. **把 CNN-LSTM 定位为对比实验，不要当唯一硬交付。**  
   当前最稳的硬结果还是 RandomForest；深度学习模型的目标应该是“跑通 + 出对比结果”，不是强行追求全面超越。

4. **把 PCAP 功能降级为“离线文件能力”而不是“真实实时流量分析能力”。**  
   当前阶段做上传、格式校验、基础解析或结果展示可以；不要把真实抓包和实时检测当硬验收项。

## 四、建议取消或降级的内容

下面这些内容，当前阶段不要继续作为硬任务分配：

1. Transformer
2. TD3 / 强化学习动态防御
3. BERT / 多模态特征融合
4. INT8 量化
5. DPDK 高性能包处理
6. 校园网实时抓包
7. “模型必须切换到 CICIDS2017 才算完成”

这些内容可以保留在“后续拓展”里，但不能再写进本阶段每日必须完成任务。

## 五、当前阶段建议目标

建议把 2026年4月20日 到 2026年5月10日 的目标压成 4 个：

1. **稳定主线闭环**  
   用现有 NSL-KDD 数据和 RandomForest 模型，把训练、推理、接口、前端展示全部打通并可重复演示。

2. **补齐对比实验**  
   跑通 CNN-LSTM 或现有 hybrid 模型中的一条深度学习路线，产出和 RandomForest 的对比表。

3. **做离线文件能力**  
   把上传接口、前端上传入口、后端基础校验和结果展示接起来，哪怕先是“上传 + 占位分析 + 后续可扩展接口”。

4. **统一文档口径**  
   把 README、界面文案、任务文档、检查报告里的夸大表述收回来，保证“写的”和“能跑的”一致。

## 九、最终建议

如果你问我一句最直接的话：

**当前最合理的分工，不是把项目往更大更炫的方向推，而是把已经做出来的主线成果先做实。**
